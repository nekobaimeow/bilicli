# v3 Algorithm Recursion → Iterative Loop

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Convert the v3 two-pointer binary-search recursion in `src/ipc/ocr/adaptive.rs` into an explicit-stack iterative loop, so the per-frame algorithm overhead (excluding OCR engine work) is < 3 seconds for the 215s 洛天依 benchmark (49 OCRs).

**Architecture:** Replace the `async fn v3_recurse(...)` recursive function (12 `Box::pin` calls) with an explicit work-stack + iteration loop. Each iteration pops one `[lo, hi]` range, runs the same 9-condition decision tree, and pushes 0-2 new ranges back onto the stack. The OCR call site (the only one in the function) is preserved verbatim. The cache, samples, budget, and ocr_calls counters stay shared. External callers (`run()` root) are unchanged except the entry point is no longer `Box::pin`'d.

**Tech Stack:** Rust (async-std-style, no extra deps). Existing types: `AdaptiveSample`, `RawDetection`, `AdaptiveConfig`, `OcrEngine`, `HashMap<i32, (PathBuf, Vec<RawDetection>)>`.

## Background

### The current overhead

实测 (215s 洛天依, 49 OCR):
- v3 wall: 43s
- Phase 0 ffmpeg: 1.5s
- OCR engine: 49 × 0.5s ≈ 24.5s
- **v3 递归纯算法 overhead = 43 − 1.5 − 24.5 = 17s** (40% of total wall time)

Per-frame v3 算法开销 = 0.35s/帧. 12 个 `Box::pin` 堆分配 + 49 个 `HashMap<i32, _>.get()` + 49 个 `(PathBuf, Vec<RawDetection>)` 解构 + 49 个 `match ocr_frame().await` + 大量 `tracing::trace!` 字符串格式化。

### Why iterative

- 递归在 Rust async 里强制 `Box::pin` (12 次堆分配)
- 每次递归 call 复制 `PathBuf` + `Vec<RawDetection>` 进栈 (实测 ~6KB/帧)
- explicit-stack + pop/push 一次内存预分配后续 O(1) push

### Acceptance criteria

1. **`v3_pure_algorithm_overhead < 3s`** for the 215s 洛天依 benchmark (49 OCR calls). Measured by subtracting OCR time + Phase 0 time from total wall time, with a stub `OcrEngine` that returns `Ok(vec![])` instantly.
2. **算法行为不变**: same 4-video E2E output: 末世 43 OCR / 别替 51 OCR / 风景 63 OCR / 洛天依 49 OCR, all `detections` lists identical to commit d6cc5bc.
3. **189/189 unit tests pass**.
4. **plan mode**: serial main-agent execution (no subagent), user wants me to drive it.

## Task 2: Port the explicit-stack algorithm into `adaptive::run`

**Why:** Once the benchmark proves the explicit-stack version is fast, port it to the production `run()` function, replacing the recursive `v3_recurse` entirely.

**Files:**
- Modify: `src/ipc/ocr/adaptive.rs:120-639` (replace Phase 1 cache + v3_recurse + root with iterative version)

**Step 2a:** Delete the inner `async fn v3_recurse(...)` and its 12 `Box::pin` call sites. Keep `async fn ocr_frame(...)` exactly as-is.

**Step 2b:** Replace `v3_recurse` and the root block with a single iterative loop:

```rust
// Phase 1: 1s-frame sampling. Use an explicit work-stack so we
// don't pay Box::pin + HashMap lookup + Vec/PathBuf clone
// overhead per recursion (the recursive version was measured
// at 17s of pure algorithm overhead for 49 OCRs).
let mut work: Vec<(i32, PathBuf, Vec<RawDetection>, i32, PathBuf, Vec<RawDetection>)> = Vec::new();

// Seed the root range. OCR idx=0 and idx=last_frame first.
let (lo_path, lo_raws, lo_cached) = match ocr_frame(0, &mut ocr_cache, engine, frames_dir, cfg).await {
    Some(r) => r, None => return (samples, ocr_calls),
};
if !lo_cached { budget_remaining -= 1; ocr_calls += 1; }
let (hi_path, hi_raws, hi_cached) = match ocr_frame(last_frame_idx, &mut ocr_cache, engine, frames_dir, cfg).await {
    Some(r) => r, None => return (samples, ocr_calls),
};
if !hi_cached { budget_remaining -= 1; ocr_calls += 1; }
work.push((0, lo_path, lo_raws, last_frame_idx, hi_path, hi_raws));

// Phase 2: explicit-stack iteration. Each iteration:
//   1. Pop a range [lo, hi] with cached raws.
//   2. Apply the same 9-condition decision tree as the recursive version.
//   3. Push 0-2 child ranges back. The OCR call for the midpoint is
//      inlined as a step before the push, with cache hit/miss tracked.
while let Some((lo_idx, lo_path, lo_raws, hi_idx, hi_path, hi_raws)) = work.pop() {
    // 1. Safety cap
    if budget_remaining == 0 { continue; }
    // 2-9: same decision tree as the recursive version.
    // ... (lift the body of v3_recurse verbatim, but with
    //      `work.push(...)` instead of `Box::pin(v3_recurse(...)).await`).
}
```

**Step 2c:** The body of the while loop is a copy-paste of the existing v3_recurse body. The ONLY changes are:
- `Box::pin(v3_recurse(...).await)` → `work.push((...))` (3-arg tuple, no return value)
- Remove `async` from the function (it's now a sync function inside an `async` outer, so just plain code)
- The 3 split paths each push 0-2 children instead of recursing

**Step 2d:** Lift each exit condition (advance / retreat / lo==hi / lo_text==hi_text / mid==lo / mid==hi / mid distinct / mid invalid) into the loop body. Each becomes:
- "advance": cache.lookup or ocr_frame(lo+1) and (hi), push (lo+1, ..., hi, ...) if valid
- "retreat": same for (lo, hi-1)
- "lo==hi": push sample
- "lo_text==hi_text": push sample, continue
- "mid==lo": ocr_frame(mid), push (lo, ..., mid, mid_raws) + (mid+1, ...)
- "mid==hi": ocr_frame(mid), push (lo, ..., mid-1, ...) + (mid, mid_raws, hi, ...)
- "mid distinct": push mid sample + (lo, ..., mid-1) + (mid+1, ..., hi)
- "mid invalid": push (lo, ..., mid-1) + (mid+1, ..., hi)
- "mid==lo (where mid_text matches lo_text)": same as above

## Pitfalls to watch for

1. **Work stack ordering**: The original `Box::pin` recursion processes children left-then-right (depth-first, left first). The explicit stack with `work.pop()` is LIFO, so the LAST pushed child runs first. For correctness, push the right child first, then the left child. The final `samples` list is then sorted by `t_sec` at the end, so order in the stack doesn't matter for output. But: if there's a side effect in execution order (e.g., `tracing::trace!` log lines), they'd reverse. This is a log-only change.

2. **Budget exhaustion mid-iteration**: The recursive version had `if *budget == 0 { return; }` at function entry. The iterative version must check this in the loop body, before any work. If budget is 0, `continue` (skip this range, don't push children).

3. **Cache hit for already-OCR'd frame**: When a child range needs `hi_idx = mid - 1` (for `mid==hi` path), the OCR for `mid-1` may already be in the cache from a previous sibling split. The `ocr_frame` function already handles cache hits and returns `(path, raws, true)`. The caller does NOT decrement budget for cache hits. Same as before.

4. **Empty/watermark-only lo path**: Original code calls `ocr_frame(hi_idx, ...)` AND `ocr_frame(lo_idx+1, ...)`. The first is redundant if `hi_idx` is unchanged (cache hit). Optimization: only OCR `hi_idx` if `hi_idx == hi_idx` (always true) — wait, the original code is correct. Keep as-is. Note: the `let _ = (new_hi_path, ...)` 3-tuple destructure is fine.

5. **primary_content_text is called twice per iteration**: Once for `lo`, once for `hi`. If `lo_raws` is empty, we return early in the `advance` branch. The double-call happens in the `lo==hi` and `lo_text==hi_text` paths (lines 358-378 of the original). In the iterative version, we should call `primary_content_text(&lo_raws)` and `primary_content_text(&hi_raws)` exactly once per iteration (cache in a local).

6. **Final samples list must be sorted by `t_sec`**: Already done at line 627 of the original. The iterative version does the same after the while loop.

7. **Test count stays at 189**: Don't add new failing tests during the refactor. The bench is `#[ignore]`. The 189 existing tests should pass without modification.
